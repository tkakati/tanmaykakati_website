FRAMES — PM System Design Framework

Copy Notion template ↗

Most effective when the problem can be structured into inputs, logic, and outputs.

F

Frame the problem

Clarify the user problem and the decision being made.

  • User problem
  • Primary user
  • Journey stage
  • Decision being made
  • Desired outcome
  • Success metric
R

Requirements & constraints

Define what must be true for the system to work.

  • Functional requirements
  • Non-functional requirements
  • Business constraints
  • UX / trust constraints
  • Assumptions
A

Actors, assets & architecture

Map system components, dependencies, and what persists.

  • Actors
  • Inputs
  • Stored state / memory
  • Dependencies
  • Core components
M

Mechanics & flow

Describe the workflow and logic that produces outputs.

  • Workflow (numbered steps)
  • Core logic / decisioning
  • Outputs
E

Exceptions & tradeoffs

Plan for edge cases, failure modes, and unavoidable tradeoffs.

  • Failure modes
  • Edge cases
  • Fallbacks / safeguards
  • Tradeoffs
  • Alternatives considered
S

Success, scale & evolution

Define success now, and how the system changes at scale.

  • Primary metric
  • Leading indicators
  • Scale considerations
  • Feedback loop
  • Future evolution

When to use FRAMES

Works well for

  • Multi-step decision systems (ranking, prioritization)
  • Feature design with clear inputs → outputs
  • Systems requiring tradeoff evaluation
  • Early-stage product design
  • Internal tools / user-driven logic systems

Not ideal for

  • Pure exploratory problems
  • Highly creative systems
  • Deep ML-first systems
  • Fully automated pipelines
  • Unstructured problem spaces
